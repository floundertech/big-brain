# Big Brain — Project Context

## The Person

- Sales engineer at Dynatrace
- ADHD — compulsive learner, high-volume info consumer, needs effortless retrieval
- Uses Plaud for call recordings (exports plain text transcripts)
- Homelab: Proxmox + Docker Compose → K8s eventually, Synology NAS
- Mobile access via VPN — responsive web UI is sufficient, no native app needed

## The Vision

The Enterprise computer. One place for everything in life — work, personal, health,
finances, homelab, learning. You ask it anything, it knows. Not a PKM tool that
becomes a write-only graveyard — a system that captures passively and retrieves
effortlessly.

## Current State (V2)

A working foundation with entity relationships. Single `docker compose up`.

- **Ingest:** paste text or upload `.txt` — Claude auto-generates title, summary, tags; extracts people and organizations
- **Browse:** list/filter entries by type (note | transcript)
- **Search:** semantic vector search (fastembed + nomic-embed-text-v1.5, local CPU)
- **Chat:** RAG chat — ask questions, Claude answers from your notes with source citations
- **Entity pages:** click a person or organization name → see all entries mentioning them; entity-focused retrieval patterns

### Stack
| Layer | Tech |
|---|---|
| LLM | Claude API (Sonnet) |
| Embeddings | fastembed + nomic-embed-text-v1.5 (local CPU, no subscription) |
| Storage | PostgreSQL + pgvector |
| Backend | FastAPI (Python) |
| Frontend | React + Tailwind (Vite) |
| Runtime | Docker Compose |

### Data model (V2)
```
entries
  id            serial PK
  created_at    timestamptz
  updated_at    timestamptz
  title         text          ← generated by Claude
  source_type   text          ← "note" | "transcript"
  raw_text      text
  summary       text          ← generated by Claude
  tags          text[]        ← generated by Claude
  embedding     vector(768)   ← nomic-embed-text-v1.5

entities
  id            serial PK
  entity_type   text          ← "person" | "organization"
  name          text          ← normalized, unique per type
  created_at    timestamptz

entry_entities (join table)
  entry_id      serial FK → entries.id (cascade delete)
  entity_id     serial FK → entities.id (cascade delete)
```

## Design Decisions

### Tag System

Tags are **typed** with prefixes. Free-text entry prompts a typed suggestion.

| Prefix | Examples | Source |
|---|---|---|
| `domain:` | `work`, `personal`, `health`, `finance`, `homelab`, `learning` | AI |
| `company:` | `dynatrace`, `acme-corp` | AI |
| `person:` | `john-smith`, `dr-patel` | AI |
| `topic:` | `apm`, `distributed-tracing`, `pricing` | AI |
| `project:` | `acme-renewal`, `homelab-rebuild` | AI + user |
| `type:` | `meeting`, `transcript`, `document`, `idea`, `reference` | AI |
| `action:` | `follow-up`, `commitment`, `decision` | AI (extracted from content) |

**Flat-to-typed UX:** User types `acme` → system suggests `company:acme-corp` → one tap to confirm or edit.

**Action item extraction:** AI scans content for commitments ("I'll send...", "we need to...", "by Friday") and tags them `action:*` + surfaces them in a separate actionable list.

### Entity Model

Entities are first-class — not just tags. They get their own pages and link to entries.

**V2 Implementation (DONE)**
- Single `entities` table with `entity_type` discriminator (`"person"` | `"organization"`)
- Unique constraint on `(entity_type, name)` prevents duplicates
- `entry_entities` join table with cascade deletes
- Extraction: second Claude API call per ingest, focused prompt for people + orgs only
- Upsert pattern: `INSERT ... ON CONFLICT DO NOTHING` (race-condition-safe)
- API: `GET /entities/` (filterable by type or entry_id), `GET /entities/{id}` (full detail with all linked entries)
- Frontend: entity pills on entry detail pages (violet = person, amber = org), clickable to entity page

**Future Entities (Post-V2)**
- **Project** — name, domain (work/personal), linked people + orgs + entries
- **Topic** — subject area, linked entries (auto-created from `topic:` tags)
- **Extended Person fields** — role, organization context, relationship notes
- **Organization fields** — type (customer/prospect/vendor/personal), metadata

**Relationship Patterns**
- Entry → belongs to many Entities (1:N via join table)
- Person → belongs to Organization(s) (future N:N)
- Project → links People + Organizations + Entries (future hub)
- Entity to Entity — eventual graph (person knows person, org acquired org, etc.)

### Research / Enrichment

**Goal:** Let Claude go get information and bring it back as a first-class entry, linkable to entities, searchable in RAG.

**Two entry points:**
- Entity page → "Enrich" button → pre-seeded prompt about that person/company
- Research page → free-text input → anything ("Dynatrace competitors", "what is eBPF")

**Source:** Claude API with web search tool (tool_use API). Live data, not just model knowledge. Web search is a provider — swap or disable independently of the rest of the stack.

**Output flow:**
1. Claude performs research with web search, returns structured markdown
2. Result saved as `Entry` with `source_type: "research"`
3. Normal enrichment pipeline runs: title, summary, tags generated
4. Entity extraction runs: people + orgs auto-linked
5. If triggered from an entity page, that entity is always linked regardless of extraction

**Entry shape:**
```
entries.source_type = "research"
entries.title       = auto-generated (e.g. "Acme Corp — Company Overview")
entries.raw_text    = Claude's research output (markdown)
entries.summary     = auto-generated
entries.tags        = auto-generated
entry_entities      = auto-extracted + seed entity (if entity-triggered)
```

**Frontend:**
- Entity detail page: "Enrich" button → loading state → new entry appears at top of entry list
- Research page: text input + "Research" button → same flow → shows generated entry on completion
- No new page types needed — result is just a normal entry

**Key constraints:**
- Web search requires Claude tool_use API — `claude-sonnet-4-6` supports this
- Rate-limit awareness: research calls will be slower + more expensive than normal enrichment
- Hallucination risk: research entries should display a "Research (AI-generated)" badge so user knows provenance
- Self-hosted constraint satisfied — no external SaaS, just Claude API calls we already make

### Retrieval Patterns (how the user actually searches)

- **Customer-related:** by Person + Company ("what did John at Acme say about pricing?")
- **Platform knowledge:** by Subject/Topic ("what do I know about distributed tracing?")
- **Projects:** by Project name ("everything related to the Acme renewal")
- **Action items:** surfaced automatically, not buried in notes

### Provider Abstractions (future-proofing)

Build thin interfaces from day one so swapping is painless:
- LLM: Claude API today → local model (Ollama) when GPU arrives
- Transcription: Plaud text today → local Whisper when moving off Plaud
- Embeddings: fastembed today → swap model without code changes
- Storage: Docker volumes today → MinIO on NAS for file storage
