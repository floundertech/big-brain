# Big Brain — Project Context

## The Person

- Sales engineer at Dynatrace
- ADHD — compulsive learner, high-volume info consumer, needs effortless retrieval
- Uses Plaud for call recordings (exports plain text transcripts)
- Homelab: Proxmox + Docker Compose → K8s eventually, Synology NAS
- Mobile access via VPN — responsive web UI is sufficient, no native app needed

## The Vision

The Enterprise computer. One place for everything in life — work, personal, health,
finances, homelab, learning. You ask it anything, it knows. Not a PKM tool that
becomes a write-only graveyard — a system that captures passively and retrieves
effortlessly.

## Current State (V2)

A working foundation with entity relationships. Single `docker compose up`.

- **Ingest:** paste text or upload `.txt` — Claude auto-generates title, summary, tags; extracts people and organizations
- **Browse:** list/filter entries by type (note | transcript)
- **Search:** semantic vector search (fastembed + nomic-embed-text-v1.5, local CPU)
- **Chat:** RAG chat — ask questions, Claude answers from your notes with source citations
- **Entity pages:** click a person or organization name → see all entries mentioning them; entity-focused retrieval patterns

### Stack
| Layer | Tech |
|---|---|
| LLM | Claude API (Sonnet) |
| Embeddings | fastembed + nomic-embed-text-v1.5 (local CPU, no subscription) |
| Storage | PostgreSQL + pgvector |
| Backend | FastAPI (Python) |
| Frontend | React + Tailwind (Vite) |
| Runtime | Docker Compose |

### Data model (V2)
```
entries
  id            serial PK
  created_at    timestamptz
  updated_at    timestamptz
  title         text          ← generated by Claude
  source_type      text          ← "note" | "transcript" | "research" | "email"
  raw_text         text
  summary          text          ← generated by Claude
  tags             text[]        ← generated by Claude
  embedding        vector(768)   ← nomic-embed-text-v1.5
  gmail_message_id text          ← nullable; Gmail dedup (email entries only)
  metadata         jsonb         ← nullable; e.g. {sources: [...urls]} for research entries

entities
  id            serial PK
  entity_type   text          ← "person" | "organization"
  name          text          ← normalized, unique per type
  created_at    timestamptz

entry_entities (join table)
  entry_id      serial FK → entries.id (cascade delete)
  entity_id     serial FK → entities.id (cascade delete)
```

## Design Decisions

### Tag System

Tags are **typed** with prefixes. Free-text entry prompts a typed suggestion.

| Prefix | Examples | Source |
|---|---|---|
| `domain:` | `work`, `personal`, `health`, `finance`, `homelab`, `learning` | AI |
| `company:` | `dynatrace`, `acme-corp` | AI |
| `person:` | `john-smith`, `dr-patel` | AI |
| `topic:` | `apm`, `distributed-tracing`, `pricing` | AI |
| `project:` | `acme-renewal`, `homelab-rebuild` | AI + user |
| `type:` | `meeting`, `transcript`, `document`, `idea`, `reference` | AI |
| `action:` | `follow-up`, `commitment`, `decision` | AI (extracted from content) |

**Flat-to-typed UX:** User types `acme` → system suggests `company:acme-corp` → one tap to confirm or edit.

**Action item extraction:** AI scans content for commitments ("I'll send...", "we need to...", "by Friday") and tags them `action:*` + surfaces them in a separate actionable list.

### Entity Model

Entities are first-class — not just tags. They get their own pages and link to entries.

**V2 Implementation (DONE)**
- Single `entities` table with `entity_type` discriminator (`"person"` | `"organization"`)
- Unique constraint on `(entity_type, name)` prevents duplicates
- `entry_entities` join table with cascade deletes
- Extraction: second Claude API call per ingest, focused prompt for people + orgs only
- Upsert pattern: `INSERT ... ON CONFLICT DO NOTHING` (race-condition-safe)
- API: `GET /entities/` (filterable by type or entry_id), `GET /entities/{id}` (full detail with all linked entries)
- Frontend: entity pills on entry detail pages (violet = person, amber = org), clickable to entity page

**Future Entities (Post-V2)**
- **Project** — name, domain (work/personal), linked people + orgs + entries
- **Topic** — subject area, linked entries (auto-created from `topic:` tags)
- **Extended Person fields** — role, organization context, relationship notes
- **Organization fields** — type (customer/prospect/vendor/personal), metadata

**Relationship Patterns**
- Entry → belongs to many Entities (1:N via join table)
- Person → belongs to Organization(s) (future N:N)
- Project → links People + Organizations + Entries (future hub)
- Entity to Entity — eventual graph (person knows person, org acquired org, etc.)

### Research / Enrichment

**Goal:** Let Claude go get information and bring it back as a first-class entry, linkable to entities, searchable in RAG.

**Two entry points:**
- Entity page → "Enrich" button → pre-seeded prompt about that person/company
- Research page → free-text input → anything ("Dynatrace competitors", "what is eBPF")

**Source:** Claude API with web search tool (tool_use API). Live data, not just model knowledge. Web search is a provider — swap or disable independently of the rest of the stack.

**Output flow:**
1. Claude performs research with web search, returns structured markdown
2. Result saved as `Entry` with `source_type: "research"`
3. Normal enrichment pipeline runs: title, summary, tags generated
4. Entity extraction runs: people + orgs auto-linked
5. If triggered from an entity page, that entity is always linked regardless of extraction

**Entry shape:**
```
entries.source_type = "research"
entries.title       = auto-generated (e.g. "Acme Corp — Company Overview")
entries.raw_text    = Claude's research output (markdown)
entries.summary     = auto-generated
entries.tags        = auto-generated
entry_entities      = auto-extracted + seed entity (if entity-triggered)
```

**Frontend:**
- Entity detail page: "Enrich" button → loading state → new entry appears at top of entry list
- Research page: text input + "Research" button → same flow → shows generated entry on completion
- No new page types needed — result is just a normal entry

**Source provenance (critical for business use):**
- Claude synthesizes from search results — it is NOT a raw search result
- Tool_use API returns the URLs Claude pulled from as part of the tool call response
- These must be stored with the entry and displayed in the UI — user needs to verify sources before acting on competitive intel, prospect data, or deal context
- Entry detail page for research entries should show a "Sources" section with all cited URLs
- Consider storing sources in a `metadata` JSON column on `entries` (nullable, only populated for research type)

**Key constraints:**
- Web search requires Claude tool_use API — `claude-sonnet-4-6` supports this
- Research calls are slower + more expensive than normal enrichment — expected, acceptable
- Research entries should display a "Research" source badge distinct from "note" and "transcript"
- Self-hosted constraint satisfied — no external SaaS, just Claude API calls we already make

### Gmail Connector

**Goal:** Zero-friction email capture. Apply a Gmail label → message appears in Big Brain automatically.

**Auth:**
- OAuth2 via Google API. One-time browser-based authorization flow that writes `gmail_token.json` to the project root (gitignored, separate from `.env`).
- Token includes a refresh token — subsequent polls authenticate silently without user interaction.
- Setup is a one-time CLI step: `python backend/scripts/gmail_auth.py` → browser opens → authorize → token saved.

**Label convention:**
- User applies `big-brain` label to any Gmail message they want ingested.
- After successful ingestion: `big-brain` label removed, `big-brain/done` label applied.
- Poller only fetches messages with `big-brain` label → naturally idempotent (processed messages are no longer visible to the poller).
- Fallback: `gmail_message_id` stored on the entry — if a message is somehow seen again, it is skipped.

**Polling:**
- Background asyncio task started in FastAPI `lifespan()` alongside the embedding model pre-warm.
- `GMAIL_POLL_INTERVAL_SECONDS` in `.env` (default: 300 / 5 min).
- If Gmail credentials not present, poller silently skips — Gmail is opt-in, not required.

**Ingest behavior:**
- Body only (plain text extracted from `text/plain` MIME part; HTML stripped as fallback).
- If the message has attachments: a note is appended to `raw_text` — *"Note: this message has N attachment(s). View the original message in Gmail."*
- Normal enrichment pipeline runs: Claude generates title, summary, tags, extracts entities.
- `source_type: "email"`, `gmail_message_id` stored in a new nullable column on `entries`.

**Entry shape:**
```
entries.source_type      = "email"
entries.title            = auto-generated by Claude
entries.raw_text         = email body + optional attachment note
entries.summary          = auto-generated
entries.tags             = auto-generated (Claude will often infer domain:work, company:*, etc.)
entries.gmail_message_id = Gmail message ID (for dedup)
entry_entities           = auto-extracted
```

**New env vars:**
| Variable | Default | Notes |
|---|---|---|
| `GMAIL_POLL_INTERVAL_SECONDS` | `300` | How often to check for new labeled messages |

**New files:**
- `backend/scripts/gmail_auth.py` — one-time OAuth flow, writes `gmail_token.json`
- `backend/app/services/gmail.py` — poll, fetch, parse, ingest
- `gmail_token.json` — gitignored, lives in project root

**Key constraints:**
- Requires a Google Cloud project with Gmail API enabled and OAuth 2.0 credentials (`credentials.json` also in project root, gitignored).
- Google OAuth requires the app to be in "test" mode or published — for personal use, test mode with the owner's account is sufficient.
- No push/webhook support planned — polling is simpler and sufficient for personal use.

### Agentic Chat

**Goal:** Chat can do things, not just answer questions. Ask Claude to research a topic or customer → it searches the web, synthesizes, confirms, and saves the result as an entry.

**Architecture:**
- Single always-on agent mode — no toggle. Claude detects intent from the prompt.
- Claude's native tool-use API (`tool_use` content blocks). No agent framework.
- System prompt defines the tool set and confirmation rules.

**Tools:**

| Tool | Scope | Cost | Confirmation required? |
|---|---|---|---|
| `search_notes(query)` | Local RAG (existing behavior) | Free | No |
| `get_entity(name)` | Local DB lookup | Free | No |
| `web_search(query, num_results)` | Tavily API | ~$0.001/call | Yes — state intent before calling |
| `save_entry(text, source_type, title)` | Local DB write | Free | Yes — show summary before saving |

**Confirmation protocol:**
- Before calling `web_search`: Claude tells the user what it's about to search. User replies to proceed (any affirmative). Claude does not search until confirmed.
- Before calling `save_entry`: Claude shows what it's about to save (title + 2-sentence summary). User confirms. Single "yes" or "save it" is sufficient.
- Local tools (`search_notes`, `get_entity`) run silently — no confirmation needed.
- This keeps costs visible and intentional without adding UI complexity.

**Web search:**
- Provider: Tavily (designed for LLM agents — returns clean extracted content, not raw HTML).
- New env var: `TAVILY_API_KEY`.
- `web_search` tool maps directly to Tavily's `/search` endpoint.

**Save behavior:**
- Saved entries: `source_type: "research"` with auto-generated title, summary, tags, entity extraction.
- Sources (Tavily result URLs) stored in `entries.metadata` JSON column (nullable) — displayed in UI under a "Sources" section on research entries.
- Auto-tagged `source:web` so research entries are filterable.

**What doesn't change:**
- Chat UI is unchanged — same input box, same message thread.
- If the user just asks a question ("what did I discuss with Sarah?"), Claude calls `search_notes` silently and answers. Behavior is identical to today.
- Agent only diverges from today's behavior when the prompt implies external action.

**New env vars:**
| Variable | Required | Notes |
|---|---|---|
| `TAVILY_API_KEY` | Only if using web search | From tavily.com |

**Key constraints:**
- `claude-sonnet-4-6` supports tool use natively — no model change needed.
- Multi-tool calls in a single turn are supported (e.g. Claude may call `web_search` 3 times then `save_entry` once).
- Agentic chat entries and normal research entries share the same `source_type: "research"` shape — no divergence in data model.

### Retrieval Patterns (how the user actually searches)

- **Customer-related:** by Person + Company ("what did John at Acme say about pricing?")
- **Platform knowledge:** by Subject/Topic ("what do I know about distributed tracing?")
- **Projects:** by Project name ("everything related to the Acme renewal")
- **Action items:** surfaced automatically, not buried in notes

### Provider Abstractions (future-proofing)

Build thin interfaces from day one so swapping is painless:
- LLM: Claude API today → local model (Ollama) when GPU arrives
- Transcription: Plaud text today → local Whisper when moving off Plaud
- Embeddings: fastembed today → swap model without code changes
- Storage: Docker volumes today → MinIO on NAS for file storage
